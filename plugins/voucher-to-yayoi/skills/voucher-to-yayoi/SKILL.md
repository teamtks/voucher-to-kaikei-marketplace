# 証憑書類 → 弥生会計インポートデータ 作成スキル

## このスキルの考え方

証憑書類(領収書・請求書)から仕訳を起票する作業では、日付・金額・取引先の読み取りに
高い精度が必要になる。ローカルのOCR(文字認識)だけでは、桁の誤読や複雑なレイアウトへの
対応に限界があることが実際の運用で確認されている。そのため、このスキルでは**証憑の
内容理解そのものはClaude自身の画像読解に任せ**、機械的に決まる部分(弥生形式への
変換ルール)だけをスクリプトに固定化している。

ただし、証憑画像には自社名(宛名)や取引先の機密情報が含まれるため、**画像をそのまま
Claudeに渡すのではなく、宛名部分だけをローカルで黒塗りしてから読み取る**という手順を
必ず踏むこと。これは情報漏洩対策として利用者と合意した重要なルールであり、省略しては
ならない。

## スキルの内容を常に最新化する(毎回、他の作業より先に自動実行すること)

Claude Desktopの「プラグイン」画面にある「同期」ボタンは、内部的に持っている
マーケットプレイスの複製を実際には更新しないことがある(実機で確認済みの不具合)。
そのままだと、GitHub上で新機能や修正を配布しても、利用者の手元では反映されない。

これを利用者に気づかせたり、設定画面を操作させたりする必要は無い。**このスキルを
使う作業を始めるたびに、他の何よりも先に、必ず以下を1回実行すること**(`python`が
まだ使えない場合は、後述の「Pythonが入っていない場合」を先に行ってから実行する):

```
python scripts\refresh_marketplace_cache.py
```

これは、マーケットプレイス経由でインストールされている場合にのみ動作し、内部の
複製を安全な方法(早送りマージのみ、強制上書きはしない)で最新化した上で、実際に
読み込まれるキャッシュフォルダの中身を最新の内容に置き換える。Claude Desktop
アプリ自身が管理する設定ファイルには一切触れない(そのため、プラグイン画面上の
「最終更新日」表示は更新されないままになるが、これは見た目だけの問題であり、
実際にスキルが使う内容は正しく最新化される。利用者から見た目の日付について
聞かれた場合は、その旨を説明すること)。マーケットプレイス経由でない場合
(このスキルフォルダを直接使っている場合)や、オフラインの場合は、何も行わずに
そのまま次に進んでよい。

## 初回セットアップ(利用者に手作業をさせず、Claude自身が毎回自動でチェック・実行すること)

このスキルを使う作業を始める前に、まず`scripts/.venv`フォルダが存在するかどうかを
確認する。**利用者に「セットアップしてください」と案内するのではなく、Claude自身が
以下を実行して、利用者には完了報告のみを行う。** 存在しない場合(初めてこのPCで
このスキルを使う場合)は、以下を実行してから作業を進める:

```
cd <このSKILLフォルダ>/scripts
python -m venv .venv
.venv\Scripts\pip install -r ../requirements.txt
```

(このセットアップ時のみインターネット接続が必要。以降は完全にローカルで動作する。
セットアップ完了後は、`scripts/.venv/Scripts/python.exe` で各スクリプトを実行する。)

Pythonそのものがこのパソコンに入っていない場合は、`python`コマンドがエラーになる。
その場合は、利用者に「Pythonをインストールしてほしい」と伝えるのではなく、Claude自身が
`winget install --id Python.Python.3.12 -e --accept-source-agreements --accept-package-agreements --silent`
のようなコマンドでインストールしてから、上記のセットアップに進むこと(これも本日の
プロジェクトで実際に行った手順であり、非技術者の利用者に手作業をさせるべきではない)。

venvのセットアップに続けて、デスクトップに案件選択用のランチャー(ショートカット)が
まだ無い場合は、以下も実行する(これも利用者に手作業をさせず、Claude自身が行うこと)。

```
scripts\.venv\Scripts\python.exe scripts\setup_launcher.py
```

これにより、デスクトップに「仕訳.TeamTKS.lnk」というショートカットが作成される。
利用者はこれをダブルクリックすることで、案件フォルダの一覧表示・新規作成・
Claude Desktopでの起動(該当フォルダを作業ディレクトリにしたCodeセッションを開く)
ができるようになる。初回起動時は案件フォルダをまとめて置く場所(ルートフォルダ)を
選ぶダイアログが表示され、以降はその場所が記憶される(このランチャーの設定は
`%APPDATA%\voucher-to-yayoi-launcher\config.json`にPCごとに保存される。スキルの
中身自体は他のPCと共通だが、この設定だけはPCごとに独立している)。

## 作業の進め方

### 1. 対象ファイルと参考データを確認する

ユーザーに次の2点を確認する(会話の中で既に分かっていれば聞き直さなくてよい):
- 処理する証憑ファイル(PDF/JPG/PNG)のパス(ランチャー経由の場合は案件フォルダの
  「証憑書類」フォルダの中身)
- その案件(顧問先)の勘定科目一覧・キーワード対応表のExcelファイルのパス(ランチャー
  経由の場合は案件フォルダの「参考資料ファイル」フォルダの中身)
  (無ければ、ひとまず参考データ無しで進めてよいか確認する)

**「参考資料ファイル」フォルダの中身(勘定科目一覧・キーワード対応表・過去の元帳など)
は、あくまで科目やルールを判定するための参照データであり、それ自体を仕訳の対象に
してはならない。** 仕訳を起票する対象は必ず「証憑書類」フォルダに入っている証憑
ファイルのみである。この2つを取り違えると、参考資料の中身から誤って仕訳を生成して
しまう事故になるため、対象ファイルと参考データのどちらがどちらか、作業前に必ず
明確に区別すること。

参考データがある場合は、内容を把握するために以下を実行して結果を読む:

```
scripts\.venv\Scripts\python.exe scripts\print_reference.py "<勘定科目一覧.xlsx>" "<キーワード対応表.xlsx>"
```

### 2. 自社を特定できる情報を黒塗りする(必ず実行すること・省略禁止)

証憑ファイル1件ごとに以下を実行する。PDFは自動でページごとの画像に変換される。

```
scripts\.venv\Scripts\python.exe scripts\mask_addressee.py "<証憑ファイル>" "<作業用フォルダ>"
```

宛名(様/御中)だけでなく、住所・電話番号・インボイス登録番号・金融機関名・支店名・
店番・口座番号・カード会社名・カード番号など、自社(依頼主)を特定できる情報を
まとめて検出し黒塗りする。実行結果には「自社を特定できる情報をN件検出し黒塗り
しました」または「検出されませんでした」と表示される。**後者の場合は、該当する
情報がそもそも無い証憑(多くのレシート等)なのか、検出に失敗しただけなのかを
一度確認すること。** 判断に迷う場合は、次のステップに進む前にユーザーに一言確認する
方が安全である。

**この後の手順(3)で読み込んでよいのは、このスクリプトが出力したマスク済み画像
(`<元ファイル名>_masked_pageN.png`)だけである。元のPDF/画像ファイルを直接
読み込んではならない。**

なお、入力がPDFの場合、このスクリプトは黒塗り前のページ画像
(`<元ファイル名>_original_pageN.png`)も同時に出力する。これは手順5のチェック
資料で人間が確認する際に使うためのものであり、**Claude自身がこの画像を読み込む
ことは無い**(ローカルで生成されるHTMLファイルに埋め込まれ、利用者自身のブラウザ
だけで表示されるものであり、外部に送信されることは無いため、黒塗りの必要が無い)。
入力がJPG/PNGの場合は、この複製は作られない(元ファイルそのものが既に存在する
ため)。

### 3. マスク済み画像を読み取り、仕訳を提案する

マスク済み画像をReadツールで開いて内容を確認し、以下を読み取る:
- 取引日付
- 金額(「合計」「ご請求金額」等、最終的に支払う/受け取る金額。小計や内訳の一部と
  混同しないよう、書類全体の文脈から総額を判断すること)
- 取引先名・内容(摘要として使う)
- 勘定科目(手順1で確認した参考データがあればそれを優先して当てはめる。無ければ
  一般的な勘定科目から妥当なものを提案し、その旨をユーザーに伝える)

**摘要欄の書き方**: 可能な限り「取引先名」「取引内容(購入品名やサービス内容)」
「インボイス登録番号」を含める。文字数の節約のため、摘要欄内のローマ字・カタカナ・
数字は半角で書く(例: `ＦamilyMart` ではなく`FamilyMart`、`１２３` ではなく`123`)。
ただし、これは摘要欄の書式であり、この規則を証憑画像の黒塗り判定や他のJSON項目に
適用してはならない。

複数の証憑をまとめて処理する場合も、1件ごとに上記を読み取ったうえで、**全件まとめて
表形式でユーザーに提示し、確認・訂正を受け付けてから次に進む。** 金額や勘定科目を
自動で確定させて先に進めてはならない — この一手間が、読み取り誤りをそのまま弥生に
取り込んでしまう事故を防ぐ最後の砦になる。

なお、証憑ごとの支払方法(現金・預金口座など、仕訳の貸方または借方の相手科目)が
文脈から不明な場合は、ユーザーに確認する。

### 4. 確定した仕訳をJSONにまとめる

ユーザーが内容を確認・訂正したら、以下の形式のJSONファイルを作成する(1証憑=1つの
voucher_id。同じvoucher_idを複数行に分ければ、借方または貸方どちらか一方を複数科目に
分割した複合仕訳になる。両側を同時に分割することはできない — その場合はvoucher_idを
分けて別々の仕訳として扱う)。

```json
{
  "start_denpyo_no": 1,
  "legs": [
    {
      "voucher_id": "20260702_familymart",
      "leg_no": 1,
      "transaction_date": "2026-07-02",
      "closing_flag": "",
      "debit":  {"account": "会議費", "sub_account": "", "department": "", "tax_category": "課対仕入込10%適格", "amount": 745, "tax_amount": 0},
      "credit": {"account": "現金", "sub_account": "", "department": "", "tax_category": "対象外", "amount": 745, "tax_amount": 0},
      "description": "FamilyMart横浜スタジアム前店　飲食料品",
      "memo": "",
      "split_side": null,
      "source_image": "<その仕訳の元になった、黒塗り前の画像ファイルパス>"
    }
  ]
}
```

`start_denpyo_no`は、既存の帳簿から続けて採番したい場合はユーザーに確認する(省略時は1)。
`transaction_date`はISO形式(YYYY-MM-DD)で書く。`closing_flag`は通常空文字列で、決算整理
仕訳の場合のみ`"本決"`とする。

`source_image`には、次の手順のチェック資料(人間が確認するためのHTML)に表示する
画像のパスを入れる。**ここは手順3でClaudeが読み取ったマスク済み画像ではなく、
黒塗り前の画像を指定すること**(JPG/PNG入力の場合は元の入力ファイルそのもの、
PDF入力の場合は手順2で追加出力された`<元ファイル名>_original_pageN.png`)。
チェック資料はユーザー自身のローカル環境だけで開くファイルであり、Claudeがこの
画像を読み込むことは無いため、黒塗りされていない元の内容を見せた方が、ユーザーが
証憑の全体を見比べて確認しやすくなる。同じvoucher_idの明細は同じ証憑から来ている
はずなので、同じ`source_image`を指定する。

### 5. チェック資料(HTML)を作成し、その場で確認・訂正してもらう

弥生形式のファイルをいきなり作る前に、**証憑画像と仕訳内容を並べて見比べられる
チェック資料を必ず作成し、ユーザーに最終確認してもらう。** これは「読み取り内容を
鵜呑みにせず、証憑と見比べて確認する」という本スキルの基本方針を、実際に見比べやすい
形にしたものであり、省略してはならない。

```
scripts\.venv\Scripts\python.exe scripts\generate_review_html.py "<手順4のJSON>" "<出力先>\仕訳チェック資料.html"
```

出力されたHTMLファイルは、ブラウザで開くと右側に仕訳の一覧、左側にクリックした仕訳の
元になった証憑画像が表示される。**このHTML画面上で、勘定科目・金額・摘要・日付などを
直接入力し直して訂正できる。** チャットに戻って伝え直す必要はない。

ユーザーにこのファイルを開いて確認してもらい、訂正があればその場で直してもらったうえで、
画面上の「変更をJSONとして保存(ダウンロード)」ボタンを押してもらう。これにより、
訂正後の内容が入力JSONと同じ形式のファイル(`<元のファイル名>_修正済み.json`)として
ダウンロードフォルダに保存される。

ユーザーから「訂正して保存した」と連絡を受けたら、そのダウンロードされたJSONファイルの
場所を確認し、**そのファイルを次の手順の入力として使う**(手順4のJSONを自分で編集し
直す必要はない)。なお、この画面での訂正は「借方金額と貸方金額を常に同じ値として同期
させる」「1伝票内の明細行の追加・削除」までは対応しているが、複合仕訳で分割している側
の判定(`split_side`)を明示的に変更する操作までは対応していないため、そこまでの修正が
必要な場合は手順4のJSONを直接編集する。

### 6. 弥生形式に変換する

チェック資料での確認・訂正が終わったら、その時点の最新のJSON(訂正があれば手順5で
ダウンロードされた`_修正済み.json`、無ければ手順4のJSON)を使って最終ファイルを作成する。

```
scripts\.venv\Scripts\python.exe scripts\generate_yayoi.py "<最新のJSON>" "<出力先>\仕訳インポート.txt"
```

貸借不一致など問題があればエラーメッセージが表示され、ファイルは出力されない。その場合は
JSONの該当箇所を修正して再実行する。成功すれば、そのまま弥生会計にインポートできる
CP932テキストファイルが出力される。

## 証憑書類からは通常出てこない複雑な仕訳(給与仕訳など)への対応

1つの伝票で「借方・貸方の両方が複数科目に分かれる」複雑な仕訳(給与の総支給額から
社会保険料・源泉税・現金支給額などを差し引く計算など)は、証憑書類(領収書・請求書)
からの自動起票では通常発生しないため、本スキルの手順1〜6(証憑の読み取りベースの
フロー)の対象外である。

ただし、ユーザーから給与仕訳のような複雑な仕訳の作成を直接依頼された場合(証憑読み取り
を経由しない、内容を直接指定してのお願い)には、`generate_yayoi.py`はこのパターンにも
対応している。手順4のJSON内で、該当伝票の各明細(leg)に`"split_side": "manual"`を
指定し、各明細の`debit`・`credit`の両方に、実際に出力したい内容を(「複合」
プレースホルダーが必要な側も含めて)そのまま指定する。この場合、自動集計や自動判定は
一切行われず、指定した内容がそのまま出力行になる。

例(役員報酬から社会保険料・源泉税・現金支給額を控除する給与仕訳。借方に実科目が
ある行、貸方に実科目がある行が混在し、「複合」の出現側が明細ごとに入れ替わる):

```json
{
  "start_denpyo_no": 501,
  "legs": [
    {
      "voucher_id": "202607_payroll",
      "leg_no": 1,
      "transaction_date": "2026-07-25",
      "debit":  {"account": "役員報酬", "sub_account": "", "department": "", "tax_category": "対象外", "amount": 500000, "tax_amount": 0},
      "credit": {"account": "複合", "sub_account": "", "department": "", "tax_category": "対象外", "amount": 500000, "tax_amount": 0},
      "description": "役員報酬　7月分",
      "split_side": "manual"
    },
    {
      "voucher_id": "202607_payroll",
      "leg_no": 2,
      "transaction_date": "2026-07-25",
      "debit":  {"account": "複合", "sub_account": "", "department": "", "tax_category": "対象外", "amount": 20000, "tax_amount": 0},
      "credit": {"account": "法定福利費", "sub_account": "", "department": "", "tax_category": "対象外", "amount": 20000, "tax_amount": 0},
      "description": "社会保険料　6月分",
      "split_side": "manual"
    },
    {
      "voucher_id": "202607_payroll",
      "leg_no": 3,
      "transaction_date": "2026-07-25",
      "debit":  {"account": "複合", "sub_account": "", "department": "", "tax_category": "対象外", "amount": 700000, "tax_amount": 0},
      "credit": {"account": "現金", "sub_account": "", "department": "", "tax_category": "対象外", "amount": 700000, "tax_amount": 0},
      "description": "給与及び通勤手当　7月分",
      "split_side": "manual"
    }
  ]
}
```

各明細の`debit.amount`と`credit.amount`は(その明細単独では)必ず一致させること
(バランスチェックの対象になる)。伝票全体としての借方合計・貸方合計は一致しなくてよい
(それがこのモードの目的である)。「複合」を書く側の税区分は`"対象外"`にする。

このモードで仕訳を作る場合も、証憑読み取りベースのフローと同様に、金額や科目の内容を
ユーザーに提示して確認を取ってから`generate_yayoi.py`を実行すること。可能であれば
手順5のチェック資料も作成して確認してもらうとより安全である(ただし現状のチェック
資料の画面編集機能は、`split_side`の値そのものを画面上で切り替える操作には対応して
いない)。
